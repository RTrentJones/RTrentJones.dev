---
title: Where Greenlight came from
date: 2026-06-22
description: I didn't set out to build a deploy framework — I set out to bring one project back from the dead, and realized I'd quietly rebuilt the same loop by hand a half-dozen times.
---

I came back to HeistMind after a few months away and found that its Supabase backend had paused itself
and then quietly archived — the free tier keeps its own clock, and mine had run out while I wasn't
paying attention. Before I could add a single feature I had to bring the whole thing back:
re-provision the database, re-wire the auth, redeploy, and then convince myself it actually worked
again, which is always the part that takes longer than it should.

Somewhere in the middle of that I realized I'd done all of this before — not for HeistMind
specifically, for everything. Every side project I'd spun up had grown its own slightly different
scaffolding around the same handful of problems: getting it deployed, noticing when a deploy was
quietly broken, and moving a good build to production without dragging a bad one along behind it. And
every one of those scaffolds had HeistMind's exact failure mode — it held up until I stopped looking
at it, and then it didn't, and I'd find out months later when some link started 404ing.

The tooling for this isn't a mystery; I've spent a decade in other people's release pipelines. But the
version I wanted for my own grab-bag of projects didn't really exist. I wanted one small harness I
could run on my laptop, that CI could run identically, and — more and more — that an agent could run
without me in the loop, so that "it passed" meant the same thing regardless of who or what was asking.
I wanted "deployed and working" to be something the system re-checked on every push and told me about,
instead of something I spot-checked after it had already broken. And I wanted all of it behind one
manifest per tool, not a bespoke heap of config for Workers here, Vercel there, a container off
somewhere else.

So I pulled those out into a thing of their own — [Greenlight](https://github.com/RTrentJones/greenlight)
— and, in the way these things go, promptly spent more time on the tool than on the project that needed
it. It's an npm package now; the site you're reading pulls it in at arm's length, and it's grown up in
public across a stack of releases, the latest of which pins the whole verify-then-promote path to a
single commit, so the thing that gets checked is the thing that ships. The machinery lives on the
[project page](/greenlight/); this isn't really a post about how it works. It's about why it exists,
which is less noble than the architecture makes it sound: I got tired of rebuilding the same loop, and
doing it with an agent finally made "one more small tool" cheap enough to actually finish. HeistMind
was the first thing it brought back; it's been quietly keeping the others alive since.

What makes it worth building, though, is that it's aimed at the problem that's actually next, not the
one we just spent two years on. Getting an agent to write the feature is close to solved; what comes
after isn't. Once ten agents on ten features cost about what one used to, the code stops being the
bottleneck and I become it. I'm the part that doesn't scale — I can start fifteen of these in an
afternoon and I cannot hold fifteen features in my head, read fifteen diffs with the attention each
deserves, and still have room left for whether any of them were worth building.

I'm not the first to land here; it's becoming the obvious shape. Boris Cherny, who leads Claude Code,
talks about the loop, and plenty of people building with agents are circling the same idea from their
own directions. Where I part from the default is which side of the loop to spend on. Almost all of the
tooling energy right now goes into the agent's context — better prompts, better retrieval, more of the
world crammed into the model. The scarcer thing to manage is the human's, because the human is the one
that got slower relative to everything else.

And my actual bet is narrower than "manage context." It's that the agent and I should be running the
_same primitives_ — the same deploy, the same verify, the same promote — so we're speaking one language
instead of two. When "it passes" means the identical thing to me and to the agent that reported it,
there's nothing to translate and nothing to re-derive: the green check is a shared fact, not a claim on
one side the other has to audit. That's what lets confidence keep pace with generation. Launch fifteen
agents and the ones that come back green are green in the only sense that was ever going to matter — the
one I defined and they ran — so my attention goes where it's actually mine: the judgment, the taste, the
is-this-the-right-thing-at-all. It shrinks both sides of the loop at once — less for the agent to carry,
less for me to hold.

Greenlight is where I'm building that bet out. It's one small harness on purpose, not a platform,
because the whole thing only works if the primitive is small enough that a person and a model can each
hold all of it. I don't think it's the final shape — I'd be suspicious of anyone certain they had that
this early — so if you're circling the same loop from your own side, or you've built your own version
and it looks nothing like mine, I'd genuinely like to see it. A fresh take on this is worth more to me
than agreement.
