---

[pg_kafka](/pg_kafka/) started with a straightforward claim: point an unmodified Kafka client at
Postgres, and the messages land in a SQL table. I built it in Rust as a `pgrx` extension, wrote a
782-test suite, and the tests passed.

But passing your own tests isn't the same as satisfying a real client on the other end of a socket.
Your test client and your server were written by the same person, on the same mental model. The
interesting failures live in the gap between "I implemented the API" and "a library nobody told about
my assumptions is happy." So to prove the claim, I built a conformance grid: on every tagged release,
it runs **kafkajs, librdkafka, kafka-python, and Sarama** against all 23 implemented APIs, real
clients against a real instance, and publishes the result.

## A proof you're allowed to disagree with

The grid is the live demo, and its value is that it isn't a green checkmark you have to trust. It's a
timestamped matrix, regenerated by CI, that says exactly which client exercised which API and what
happened. Right now all four clients pass every API they support — but the honest part is the cells
that _aren't_ green.

kafka-python doesn't implement Kafka transactions. So its transaction cells —`InitProducerId`,
`AddPartitionsToTxn`, `EndTxn`, the rest — are marked **N/A**, not quietly passed. That distinction is
the whole ethic of the project. A grid that paints "not exercised" the same color as "verified
working" is worse than no grid; it launders ignorance into confidence. The N/A is me refusing to take
credit for a test that never ran.

## Naming the limits out loud

The grid proves what's covered. A claim is only worth something if it also tells you where it stops,
so pg_kafka ships a
[`PROTOCOL_DEVIATIONS.md`](https://github.com/RTrentJones/pg_kafka/blob/main/docs/PROTOCOL_DEVIATIONS.md)
that spells out where it diverges from a real broker. A few of them:

- **Fetch sessions** aren't tracked — it always uses session id 0, so a client relying on incremental
  fetch sessions won't get them.
- **ListOffsets** supports earliest and latest, but timestamp-based offset lookup returns
  `UNSUPPORTED_VERSION` — there's no time index behind it.
- **Null-key produce** uses random partition assignment instead of Kafka's sticky partitioner.

The implemented surface is about 46% of the protocol. Writing that number down, next to a grid that
could contradict me, is the point — not a disclaimer to bury.

## Performance, against honest anchors

A throughput number alone is theater, so the benchmark reports against two anchors: a raw `INSERT`
**floor** (the storage layer with no protocol on top) and a real Kafka broker **ceiling**. Batched
produce runs around **8,400 msgs/sec** — comfortably above the ~2,600 raw-`INSERT` floor and the
~1,300 single-broker ceiling in the same co-located setup — while unbatched produce sits far lower, as
it should. The framing matters more than the figure: you can see what the protocol layer costs and
what batching buys, instead of taking one number on faith.

## The thesis

The honest version of a conformance claim is the one that publishes its own limits. The grid can fail
in public, the N/A cells admit what wasn't tested, and the deviations doc names where the
implementation ends. That's not hedging — it's the only kind of proof worth shipping.
