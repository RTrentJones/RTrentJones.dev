---
title: 'pg_kafka: taking "Postgres is all you need" literally enough to test it'
date: 2026-07-08
description: I agreed with the meme and couldn't ignore the objection to it — so I built the pg_kafka extension someone kept asking for, and let four Kafka clients it had never met try to prove it wrong.
---

[pg_kafka](/pg_kafka/) started with an argument I mostly agreed with and one objection I couldn't put
down.

The argument is the [familiar one](https://dev.to/shayy/postgres-is-all-you-need-3pgb): most products
don't need a distributed-systems museum on day one. If Postgres already holds your relational data, it
can probably be your job queue (`SELECT … FOR UPDATE SKIP LOCKED`), your vector store (`pgvector`), your
cache, and enough of your event plumbing that reaching for Redis and Kafka and Pinecone up front isn't
maturity — it's just complexity you volunteered for. Stanislav Kozlovski took the Kafka corner of that
specifically, [benchmarked Postgres as a pub/sub log](https://topicpartition.io/blog/postgres-pubsub-queue-benchmarks),
and landed on a line that stuck with me: _"I don't want you to rebuild Kafka, but I want someone to do
it."_

The objection is just as fair, and [Gunnar Morling made it
well](https://www.morling.dev/blog/you-dont-need-kafka-just-use-postgres-considered-harmful/): Kafka
isn't only a place to put messages. It's a protocol, a decade of client libraries, consumer groups,
offsets, retention, and a pile of operational assumptions that all expect a Kafka-shaped thing on the
other end of the socket. Swap in "just `INSERT` into a table" and you haven't replaced Kafka — you've
handed every app a migration project and called it simplicity.

Put those two together and the interesting question gets much narrower than "can Postgres replace
Kafka." It's: could Postgres be the broker _without asking the clients to care_? Kozlovski had already
sketched the answer — [on
LinkedIn](https://www.linkedin.com/posts/stanislavkozlovski_github-tansu-iotansu-apache-kafka-compatible-activity-7394087097726468096-r-T9),
bundle it into Postgres as an extension that runs a real Kafka TCP server from a background worker, with
the SQL-native and wire-protocol sides sharing the same tables — which is, give or take, what pg_kafka
turned out to be. And [elsewhere](https://blog.2minutestreaming.com/p/just-use-postgres) he said the
quiet part out loud: "technically, the storage engine is good enough… the UX isn't there," and someone
should create a `pg_kafka` extension. I'd already started one. It's hard to resist a spec that arrives
with its own name.

So that's what pg_kafka is: a PostgreSQL extension, written in Rust with `pgrx`, where a background
worker binds `:9092`, speaks the Kafka wire protocol, and lands every message in an ordinary SQL table.
The clients still speak Kafka. The storage engine is Postgres. The contract is the protocol, not an API
you import — which is the whole point, because it means the client never has to find out. It's a
portfolio project, not a broker I'd page anyone about: 23 of Kafka's ~50 APIs, single node, no
clustering. But the claim it makes is real, and — more to the point — it's falsifiable.

That last part is the part I actually care about. Getting a demo to work is cheap; anyone can produce one
message and screenshot it. The honest version of this claim isn't a demo, it's a test somebody who isn't
me could run and watch fail. My own suite was 782 tests deep and green, and it proved almost nothing — my
test client and my server came out of the same head, on the same assumptions, so passing them only means
I'm consistent with myself. The failures that matter live in the gap between "I implemented the API" and
"a library nobody told about my mental model is happy."

So the proof isn't my tests. On every tagged release, CI points four real, independently-written Kafka
clients — **kafkajs, librdkafka, kafka-python, and Sarama** — at a live instance, runs them across every
implemented API, and publishes the result as a timestamped grid. Not a green badge you take on faith: a
matrix that says which client exercised which API and what happened, regenerated where you can watch it.
The honest cells are the ones that _aren't_ green — kafka-python implements no transactions, so its
transaction APIs are marked **N/A**, not quietly passed. A grid that colors "never tested" the same as
"verified" launders ignorance into confidence, and refusing to do that was the entire reason to build it.

It earned its keep the first time it ran. Sarama — the Go client — couldn't even connect. I'd tightened
version negotiation so pg_kafka strictly rejected any request past what it advertised, which sounds
correct right up until it isn't: Sarama pins a broker version and doesn't down-negotiate, so it opens
with a Metadata request at v10, take it or leave it. pg_kafka advertised v9, left it, and handed the
rejection back in an old error format Sarama couldn't parse — so it read that 57-byte frame as the v10
response it expected, walked off the end of the buffer, and panicked. One client, one assumption I'd
never have thought to question, because I'd only ever tested myself against myself. The fix was a single
line of intent: advertise the version I actually serve.

Then there's the part Morling was right about. The grid proves what's covered; a claim is only worth
something if it also says where it stops. So pg_kafka ships a
[`PROTOCOL_DEVIATIONS.md`](https://github.com/RTrentJones/pg_kafka/blob/main/docs/PROTOCOL_DEVIATIONS.md)
that names the seams out loud — no fetch sessions, no timestamp-based offset lookup, random partition
assignment where Kafka would be sticky, no wire auth at all, one node. The implemented surface is about
46% of the protocol, and most of what's missing is exactly the ground Morling said Postgres gives up:
retention, real fault tolerance, the streaming semantics a log is actually for. He's right, and pg_kafka
doesn't argue otherwise. Writing that number down, next to a grid that could contradict me, is the point
— not a disclaimer to bury at the bottom.

There's a benchmark too, and it's deliberately not a leaderboard. pg_kafka runs next to a raw `INSERT`
floor and a real Kafka broker ceiling under one single-client, synchronous harness, so you read the table
for _shape_ — what the protocol layer costs, what batching buys — not a headline number I'd then have to
keep true. The [live table](/pg_kafka/) carries the current figures, regenerated by CI on each release;
I'd rather point at something that stays honest on its own than freeze a number into prose that quietly
goes stale.

That's the line I wanted to walk: playful enough to take a meme literally, serious enough to publish
where the meme stops. "Postgres is all you need" is usually an argument for _less_ — fewer systems, less
to run. pg_kafka is that argument pushed until it disappears: the client keeps its Kafka, Postgres
quietly becomes the thing underneath, and the only way you'd know is to read the grid. Which you can —
including the cells where it doesn't hold.
