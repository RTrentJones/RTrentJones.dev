> BAMCP gives an AI assistant eyes on the genome. It's a Model Context Protocol server that browses
> BAM/CRAM alignment files and renders **IGV-style read pileups inline in the conversation** — so a
> model reasons over the actual evidence at a locus instead of guessing from a text summary. It
> exposes 11 genomic tools, including ClinVar and gnomAD lookups, that ground variant interpretation
> in real data.

This is the thread I care about most. Before Amazon I was the second engineer at
[Fulgent Genetics](https://www.fulgentgenetics.com/), building the variant-curation platform
clinical scientists used to call variants. BAMCP is that domain pointed at agents: the bottleneck in
applying an LLM to genomics is that it can't _look_ at the alignment, so it confabulates. Give it the
pileup and the population frequencies, and the work changes from plausible-sounding to checkable.

## What it does

- **Sees the reads.** `visualize_region` renders aligned reads as an HTML5 Canvas pileup —
  color-coded mismatches, indels, and soft-clips, packed to handle 1,000–10,000+ reads per region —
  delivered inline via the **MCP Apps Extension** (an `text/html+mcp` resource), so it pans and zooms
  in the chat rather than dumping text.
- **Grounds the call.** `get_variants`, `get_coverage`, `lookup_clinvar`, `lookup_gnomad`,
  `get_variant_curation_summary`, and `search_gene` pull non-reference alleles, depth, clinical
  significance, and population frequency from real sources — the inputs a curator actually weighs.

## The workflow

The path mirrors how a curator actually works: start from a gene, locus, or candidate variant; search
the gene; look at the aligned reads; check coverage; surface the non-reference alleles; and pull
ClinVar and gnomAD context. What comes back isn't a verdict — it's a trail of evidence the model _and_
the human are looking at together, so a summary can be checked against the reads instead of taken on
faith.

## What it does not decide

BAMCP is not a clinical decision system. It doesn't make a final pathogenicity call or stand in for a
curator; its job is to put the right evidence in front of the model and the person at the same time.
That framing is also the guardrail against the failure modes a text-only agent invites — hallucinating
evidence that isn't in the reads, ignoring low coverage, over-reading a ClinVar entry, missing allele
balance, or treating a population frequency as more decisive than it is. With the pileup on screen,
those become checkable rather than plausible.

## The stack

- **Python** (`pysam` over htslib, **FastMCP**) — BAM/CRAM, including indexed remote files.
- A **Model Context Protocol** server (streamable HTTP), OAuth-gated.
- Runs on an **Oracle Cloud Always-Free Ampere A1 Container Instance** — the whole network (VCN +
  subnet) is Terraform, fronted by a Cloudflare tunnel; the only manual input is the API key.
- Built + pushed as a container image by its own CI; "deploy" is a restart that re-pulls.

## What I find interesting

- **Liveness as a feature** — a Cloudflare Worker cron health-checks the box and, on an outage,
  fires a `repository_dispatch` that re-applies + redeploys + verifies it. Recover-on-alert keeps it
  on the free tier (no pay-as-you-go).
- **Protocol-level verification** — the deploy gate speaks MCP: initialize → `tools/list` → call a
  tool and assert the shape → assert auth rejection. A drift-guard pins the exact tool set, so a
  dropped or renamed tool fails the deploy instead of silently shipping.
