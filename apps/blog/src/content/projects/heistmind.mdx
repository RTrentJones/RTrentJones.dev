> HeistMind is a character and crew manager for **Forged-in-the-Dark** tabletop RPGs — the
> mechanical backbone for asynchronous, play-by-post campaigns. Game masters run custom rulesets,
> crews, factions, and progress clocks; players build characters and track stress, harm, and XP;
> everyone joins a game by code and shares one campaign log. A Discord bot handles at-the-table
> actions without leaving chat, and the app enforces the rules, so every action it offers is legal
> for your ruleset.

## Why it exists

HeistMind is the tool I wanted and couldn't find. D&D has a deep bench of play aids; when I went
looking for somewhere to start with Blades in the Dark, there was nothing equivalent — so I built the
companion I'd have wanted on day one.

Play-by-post campaigns are collaborative software without the software: characters, crews, factions,
clocks, harm, XP, and a table's worth of house rules, spread across Discord threads, pinned messages,
and spreadsheets. HeistMind puts that state in one place and only offers actions that are legal for
the ruleset in play.

It's built to take what you want and leave the rest — a lightweight character tracker for a live
table, or the full campaign layer (crews, factions, clocks, a shared log, join-by-code) for an async,
play-by-post game on Discord. As much or as little as your table needs.

## What makes it hard

- **Two real roles.** GM and Player see and do different things; the interesting bugs live in the
  permission boundary between them.
- **Custom rulesets without conditional soup.** Forged-in-the-Dark is a family of games, not one — the
  domain model has to bend to a group's variant instead of collapsing into a pile of `if`s.
- **Auth you can actually test.** Those journeys only matter once you're authenticated — the part a
  unit test skips and a smoke test fakes — so the boundary has to be real _and_ verifiable (Discord
  OAuth, Supabase row-level security, and the deploy gate that drives it, below).
- **A bot that acts outside RLS.** The Discord bot verifies each interaction (Ed25519) and then acts
  through the service-role client, which bypasses row-level security — so the engine has to re-assert
  ownership itself, in code, rather than leaning on the database to enforce it.

## The stack

- **Next.js 15 / React 19 / TypeScript / Tailwind 4** on **Vercel** (git-integration deploys), with
  **Zustand** for local domain state and **TanStack Query** as the data seam over the Supabase
  repositories.
- **Supabase** for Postgres (with row-level security) + auth.
- **Discord** two ways: **OAuth** as the sign-in method, plus a first-party **bot** — an
  Ed25519-verified slash-command handler at `/api/discord` for actions taken without leaving chat.
  Both are declared in Terraform, so the auth and app config are managed as code, not clicked in a
  dashboard.
- A shared `@heist-mind/ui` library (Radix primitives + Tailwind) — the app's own design system.

## What I find interesting

It's the first thing I built greenfield with an agent, and it's had two lives: a first pass with
Cline in 2025, then a near-total rebuild with Claude Code in 2026.

That rebuild is also where Greenlight came from. Coming back to HeistMind after a while, I found
Supabase had paused and then archived the original backend, so I had to stand the whole thing up
again — dev loop and all. Recreating that deploy → verify → release loop by hand, for a single
project, is what made me realize every project I had wanted the same primitives. That became
[Greenlight](https://github.com/RTrentJones/greenlight).

So HeistMind rides the loop it kicked off:

- **A prod ship-gate** — branch protection requires CI to pass, so the promotion path only advances
  CI-green commits.
- **An authenticated deploy gate** — every preview deploy runs a Playwright suite that injects real
  GM/Player sessions via the Supabase service-role key (no scripting Discord's consent screen), so
  authenticated journeys are verified against each deploy, not just unit-tested.
- The whole thing is one manifest entry; the infra (domains, env, auth, keepalive) is Terraform the
  CLI emits.

Prod itself is behind a pre-launch gate right now while I finish a security pass — the full app runs
on beta and preview today, and the public launch flips on once that's done.
