> A native PostgreSQL extension (Rust / `pgrx`) that embeds the **Kafka wire protocol** in the
> database runtime. Point an unmodified Kafka client at Postgres on `:9092`, produce and consume —
> and the messages are sitting in a SQL table. 23 Kafka APIs, consumer groups, idempotent +
> transactional producers, shadow-forwarding to an external Kafka cluster, 782 tests.

## Why this exists

pg_kafka started as a response to the ["Postgres is all you
need"](https://dev.to/shayy/postgres-is-all-you-need-3pgb) discourse — not the strawman version, the
reasonable one. The reasonable version isn't "Postgres should replace every specialized system." It's:
before you add another moving part, check whether Postgres already covers the shape of the problem. That
held up for queues, search, vectors, and key-value. Kafka was the harder case, and the people
[benchmarking it](https://topicpartition.io/blog/postgres-pubsub-queue-benchmarks)
[knew it](https://blog.2minutestreaming.com/p/just-use-postgres): Postgres might be a good enough storage
engine, but Kafka's moat was never only throughput — it's the protocol, the clients, and the ecosystem.
Make every app learn a new SQL-based queue API and you didn't replace Kafka; you started a migration.

So pg_kafka asks a narrower, more testable question. Not "can Postgres replace Kafka at Kafka's scale,"
and not "should you run this in production" — just: **can an unmodified Kafka client connect to Postgres,
produce, consume, coordinate a group, and drive the transactional producer APIs without ever knowing the
broker is a database extension?** That's the claim the rest of this page tries to make falsifiable.

## The demo is the conformance report

I built the first version of pg_kafka in about three weeks with Claude Code, then didn't touch it for
months. It was the first time the deploy-and-verify loop really clicked for me — because a Kafka
implementation comes with its own test harness built in. The whole claim is that unmodified clients
just work, so the clients _are_ the check: if kafkajs and librdkafka can produce and consume without
knowing they're talking to Postgres, it works, and if they can't, it doesn't. I didn't have to invent
a way to verify it.

I came back to it while building Greenlight, where the awkward part was that an extension has nothing
to _show_ — there's no button to click and watch it go. So the demo became the evidence: run the real
clients on every tagged release, grade the coverage, and publish the grid and the benchmarks (that's
the [Tracer](/tracer/) thread, still a prototype). Making "does it work" reportable is also what let
me hand it back to agents and let them iterate without me watching every change.

## The thesis, proven: unmodified clients just work

The claim isn't "complete Kafka" — it's the narrow, testable one from the top of the page, and its one
virtue is that it can be proven directly instead of asserted. So prove it — run a real conformance pass
for **kafkajs, librdkafka, kafka-python, and Sarama** against the 23 implemented APIs and render the
grid. This is the live demo: the same Tracer pattern (continuously-verified, timestamped), applied
to protocol coverage instead of eval pass-rates. Where a client genuinely doesn't exercise an API —
kafka-python has no transaction support — the cell is marked **N/A**, not quietly passed. The grid
shows what's covered _and_ where it stops.

**At a glance:** four client libraries, 23 Kafka APIs, every supported cell passing — with the
transaction cells marked N/A for kafka-python, which doesn't expose them.

## What the verification loop caught

The proof that the reportable version was worth building is a bug the grid caught that I never would
have. My suite was 782 tests deep and green, but I'd only ever run it against one client — my own
mental model in different hats. The four-client grid ran Sarama, the Go one, and it couldn't even
connect.

I'd recently tightened version negotiation so pg_kafka strictly rejected any request whose version ran
past what it advertised, which sounds correct. But Sarama is pinned to a broker version and doesn't
down-negotiate like the others; it opens with a Metadata request at v10, take it or leave it. pg_kafka
advertised v9, left it, and handed back the rejection as an old-format error frame a strict client
can't parse — so Sarama read that 57-byte error as the v10 response it expected, walked off the end of
the buffer, and panicked, taking the rest of its column down with it. The fix was one line's worth of
intent — advertise the version I actually serve. It had been sitting there the whole time, invisible,
because I'd only ever tested myself against myself.

## Performance, in context

Co-located methodology: warm-up excluded, a fixed 1 KB payload, batched vs unbatched, reported as
p50/p99 and msgs/sec — run next to two reference points under the _same_ single-client, synchronous
harness: a raw-`INSERT` (the storage layer with no protocol) and a real Kafka broker. They're a
like-for-like baseline, not each system's tuned throughput — in a single-client loop batching
dominates, so read the table for _shape_ (what the protocol layer costs, what batching buys), not as
a leaderboard. Normalized scores also flow into
[Tracer](https://tracer.rtrentjones.dev/runs?tool=pg_kafka) — the eval dashboard, with pg_kafka
wired in first — on each tagged release, so regressions show up over time.

## Shadow-mode forwarding, verified

pg_kafka can **dual-write** to an external Kafka cluster while still storing locally — the migration /
validation / hybrid path (produce to Postgres, forward to a real broker, per-topic and percentage-sampled,
transaction-aware). The interesting part isn't that it forwards; it's that the forwarding is proven the
same way everything else here is: each scenario produces to pg_kafka, drives the live outbox path, then
**independently consumes the real broker** to confirm what actually landed — regenerated in CI, timestamped.
It doubles as a regression guard: the counts are identical before and after removing the dead pre-outbox
producer, which is objective proof the durable outbox is what does the work.

## The duality, in one session

`kcat -P` produces over the Kafka wire protocol; `kcat -C` consumes it back; then a `SELECT` over
`kafka.messages` shows the very same row in Postgres. That's the whole idea — the broker and the
table are one store. Want the real thing instead of a recording of it? The repo ships a
`.devcontainer`, so one click gives you a running instance.

## What I find interesting

- **The protocol is the product.** Binary Kafka frames are parsed and answered from inside the
  Postgres runtime — produce/fetch with `RecordBatch v2`, a full consumer-group coordinator
  (find-coordinator → join → sync → heartbeat → rebalance), and EOS transactions with
  read-committed isolation. The 782-test suite is what keeps all of that honest.
- **Evidence over assertion.** Nothing here asks you to take the README's word for it. The grid, the
  numbers, and the session are regenerated by CI and timestamped — the match of modality to what the
  project actually proves: an _implementation_ is judged by conformance and performance, not by a
  frontend.
- **It names its own limits.** The implemented surface is ~46% of the protocol, and
  [`PROTOCOL_DEVIATIONS.md`](https://github.com/RTrentJones/pg_kafka/blob/main/docs/PROTOCOL_DEVIATIONS.md)
  spells out exactly where it diverges from a real broker — fetch sessions, timestamp-based offset
  lookup, sticky partitioning. A conformance claim is only worth anything if it tells you where it
  stops.
