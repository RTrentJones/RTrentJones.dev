> A native PostgreSQL extension (Rust / `pgrx`) that embeds the **Kafka wire protocol** in the
> database runtime. Point an unmodified Kafka client at Postgres on `:9092`, produce and consume —
> and the messages are sitting in a SQL table. 23 Kafka APIs, consumer groups, idempotent +
> transactional producers, shadow-forwarding to an external Kafka cluster, 782 tests.

## Why there's no "live demo" here

A produce/consume button would be glue code, and it would make a wire-protocol-in-Postgres
implementation read as a toy — the work would go into its least impressive part while the actual
signal (the protocol layer, consumer groups, transactions, the conformance suite) sits in the
background. For an implementation, "live" shouldn't mean a hosted service with a frontend. It should
mean **continuously-verified evidence** — which is exactly what runs ephemerally in CI: spin the
`docker-compose` up in a runner, run the checks against real clients, publish the results, tear it
down. No public instance, no abuse surface, no bespoke UI. Three artifacts, each pointing at systems
depth instead of away from it.

## The thesis, proven: unmodified clients just work

The claim is "any Kafka client works, no code changes." So prove it — run a real conformance pass
for **kafkajs, librdkafka, kafka-python, and Sarama** against the 23 implemented APIs and render the
grid. This is the live demo: the same Tracer pattern (continuously-verified, timestamped), applied
to protocol coverage instead of eval pass-rates. Where a client genuinely doesn't exercise an API —
kafka-python has no transaction support — the cell is marked **N/A**, not quietly passed. The grid
shows what's covered _and_ where it stops.

## Performance, in context

Co-located methodology: warm-up excluded, multiple message sizes, batched vs unbatched, reported as
p50/p99 and msgs/sec — measured against two honest anchors, a raw-`INSERT` **floor** (the storage
layer with no protocol) and a real Kafka broker **ceiling**. Normalized scores flow into
[Tracer](https://tracer.rtrentjones.dev/runs?tool=pg_kafka) on each tagged release, so the trend —
and any regression — shows up over time for free.

## Shadow-mode forwarding, verified

pg_kafka can **dual-write** to an external Kafka cluster while still storing locally — the migration /
validation / hybrid path (produce to Postgres, forward to a real broker, per-topic and percentage-sampled,
transaction-aware). The interesting part isn't that it forwards; it's that the forwarding is proven the
same way everything else here is: each scenario produces to pg_kafka, drives the live outbox path, then
**independently consumes the real broker** to confirm what actually landed — regenerated in CI, timestamped.
It doubles as a regression guard: the counts are identical before and after removing the dead pre-outbox
producer, which is objective proof the durable outbox is what does the work.

## The duality, in one session

`kcat -P` produces over the Kafka wire protocol; `kcat -C` consumes it back; then a `SELECT` over
`kafka.messages` shows the very same row in Postgres. That's the whole idea — the broker and the
table are one store. Want the real thing instead of a recording of it? The repo ships a
`.devcontainer`, so one click gives you a running instance.

## What I find interesting

- **The protocol is the product.** Binary Kafka frames are parsed and answered from inside the
  Postgres runtime — produce/fetch with `RecordBatch v2`, a full consumer-group coordinator
  (find-coordinator → join → sync → heartbeat → rebalance), and EOS transactions with
  read-committed isolation. The 782-test suite is what keeps all of that honest.
- **Evidence over assertion.** Nothing here asks you to take the README's word for it. The grid, the
  numbers, and the session are regenerated by CI and timestamped — the match of modality to what the
  project actually proves: an _implementation_ is judged by conformance and performance, not by a
  frontend.
- **It names its own limits.** The implemented surface is ~46% of the protocol, and
  [`PROTOCOL_DEVIATIONS.md`](https://github.com/RTrentJones/pg_kafka/blob/main/docs/PROTOCOL_DEVIATIONS.md)
  spells out exactly where it diverges from a real broker — fetch sessions, timestamp-based offset
  lookup, sticky partitioning. A conformance claim is only worth anything if it tells you where it
  stops.
